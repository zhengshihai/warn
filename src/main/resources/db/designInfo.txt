#在晚归记录表（late_return）中
    -status 字段：
    这是记录的整体状态
    PENDING（待处理）：表示这条晚归记录刚刚被记录，还没有被宿管或管理员处理
    PROCESSED（已处理）：表示这条晚归记录已经被相关人员查看并处理完成
    主要用于快速筛选哪些记录需要处理，哪些已经处理过

    -process_status 字段：
    Apply to designInfo.t...
    这是具体的处理结果状态
    PENDING（待处理）：还未做出处理决定
    APPROVED（已通过）：晚归理由被认可，比如参加学校活动、图书馆学习等正当理由
    REJECTED（已驳回）：晚归理由不被接受，可能需要进一步的教育或处理
    用于记录对晚归原因的审核结果

   - process_remark 字段：
    Apply to designInfo.t...
    用于记录处理人员的具体处理意见
    可以记录为什么通过或驳回
    可以记录对学生的具体要求或建议
    便于后续跟踪和查证

    -实际使用场景示例：
    学生晚归后：
    status = 'PENDING'
    process_status = 'PENDING'
    process_remark = NULL
    宿管审核通过正当理由的情况：
    status = 'PROCESSED'
    process_status = 'APPROVED'
    process_remark = '学生因参加学校组织的文艺汇演晚归，已核实属实'
    宿管驳回无正当理由的情况：
    status = 'PROCESSED'
    process_status = 'REJECTED'
    process_remark = '无正当理由，建议加强时间管理意识，避免再次晚归'

#system_rule和 warning_rule 这两张表的作用

-system_rule（系统规则表）：
    这是一个系统级的配置表，用于存储系统运行所需的各种参数配置，比如：
    晚归时间点设置：LATE_RETURN_TIME = 23:00:00
    连续晚归阈值：MAX_CONTINUOUS_LATE = 3
    各类通知模板：

    特点：
        采用key-value形式存储
        可以随时修改系统参数而不需要重启
        适合存储全局性的配置项

-warning_rule（预警规则表）：
    这是一个业务级的规则表，用于配置具体的预警策略，比如：
    一周三次预警：
    连续晚归预警：

    特点：
        每条记录代表一个完整的预警规则
        可以设置启用/禁用状态
        支持多种预警条件组合

-两者的主要区别：
    使用场景：
    system_rule：系统运行参数配置，如时间点、模板等
    warning_rule：业务预警规则配置，如预警条件、通知方式等
    数据结构：
    system_rule：简单的key-value结构
    warning_rule：复杂的规则结构，包含多个业务字段
    使用方式：
    system_rule：系统启动时加载，运行时可修改
    warning_rule：预警任务执行时动态查询使用
    配置粒度：
    system_rule：系统级别，全局生效
    warning_rule：规则级别，可以针对不同情况配置不同规则

5 班级管理员端


定位信息存储业务流程（含重试与补偿机制）
1. 实时写入Redis
学生端/前端上传定位信息到后端。
后端首先将定位信息写入Redis（如GEO结构、ZSet等），保证教师端实时查询轨迹的高可用和低延迟。
2. 异步写入MySQL
写入Redis后，异步尝试将定位信息写入MySQL，以保证数据最终持久化。
异步写入MySQL时，最多重试3次（或你设定的次数），每次失败会记录日志。
3. RocketMQ补偿队列
如果异步写MySQL连续重试3次仍然失败，则将该定位信息发送到RocketMQ的补偿Topic（如alarm-location-compensate）。
补偿消息内容为定位信息对象（含所有字段），可额外带上retryCount字段。
4. RocketMQ补偿消费
有专门的RocketMQ Consumer监听补偿Topic。
Consumer收到补偿消息后，再次尝试写入MySQL。
如果写入成功，消息消费完成。
如果写入失败，RocketMQ会自动重试消费，直到成功或达到最大重试次数（RocketMQ可配置）。
5. Redis List兜底
如果RocketMQ不可用（如网络故障、服务不可达等），则将补偿消息写入Redis List（如location:compensate:redis），并带上retryCount字段（初始为0）。
这样可以保证即使消息队列不可用，数据也不会丢失。
6. Redis List定时补偿任务
有定时任务（如每分钟）扫描Redis List，每次最多处理10条。
取出一条后，尝试再次投递到RocketMQ补偿Topic。
如果投递成功，该条数据流程结束。
如果投递失败，解析并累加retryCount字段：
如果retryCount未达到最大重试次数（如10次），则retryCount+1后重新入队，等待下次定时任务再补偿。
如果retryCount已达到最大重试次数，则将该数据写入死信队列（如location:compensate:dead），并记录日志或告警，便于人工后续处理。
7. 死信队列与人工处理
死信队列中的数据需要人工巡查、导出或通过管理后台进行补偿或告警。
可以定期监控死信队列长度，超阈值自动告警。

手工/模拟异常测试建议
2.1 模拟MySQL异常
在异步写MySQL的代码里临时加一个throw new RuntimeException("mock db error")，观察是否能进入补偿队列。
检查补偿队列、死信队列是否有数据。
2.2 关闭RocketMQ服务
停掉RocketMQ服务，观察补偿消息是否写入Redis List兜底。
恢复RocketMQ后，观察定时任务是否能自动补偿成功。
2.3 造数据测试死信队列
手动将一条数据的retryCount设为9，模拟补偿失败，看是否能进入死信队列。
2.4 日志与监控
检查日志输出，确认每个环节的异常、重试、死信等都有明确日志。
可以临时加System.out.println或logger.info辅助观察。




sessionLoginUUID
由后端在用户登录成功后生成（如 UUID）。
后端将其保存在 Spring Session（即服务端 session）中。
后端将其返回给前端（可以放在响应体或响应头）。
requestLoginUUID
前端收到 sessionLoginUUID 后，本地保存（如 localStorage）。
之后每次请求业务接口时，前端会把这个 loginUUID 作为 requestLoginUUID 发送给后端（通常放在请求头）。
校验流程
后端拦截器拿到 requestLoginUUID（来自请求头）和 sessionLoginUUID（来自 session）。
只有两者一致时，才允许访问业务接口，否则强制下线或重定向登录。

-- 通知表
CREATE TABLE IF NOT EXISTS notification (
        id INT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
        notice_id VARCHAR(32) NOT NULL COMMENT '通知ID',
        title VARCHAR(200) NOT NULL COMMENT '通知标题',
        content TEXT NOT NULL COMMENT '通知内容',
        notice_type VARCHAR(50) COMMENT '通知类型（系统通知/晚归通知/预警通知等）',
        target_type VARCHAR(20) COMMENT
        '目标类型（student, dormitorymanager, counselor, classteacher, dean）',
        target_id VARCHAR(50) COMMENT '目标ID（特定用户的唯一标识，如学号、工号等）',
        target_scope VARCHAR(20) NOT NULL DEFAULT 'allUsers'
               COMMENT '接受通知的对象范围  allUsers, specialRole, specialUser',
        create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
        update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
        KEY idx_target (target_type, target_id),
        UNIQUE KEY uk_notice_id (notice_id),
        KEY idx_type (notice_type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='通知记录表';

CREATE TABLE IF NOT EXISTS notification_receiver (

   target_type(这里是根据用户的具体角色决定，例如学生就是STUDENT)，
   target_id(特定某个用户的业务标识id）,
   target_scope（发给某个特定用户的消息，设为SPECIAL_USER）,
   create_time(new Date()), update_time(new Date())。

         在notification插入一条数据后，我们还要在notification_receiver插入一条数据，
   包括id, notice_id(和notification表的notice_id相同，是业务标识id)，
   receiver_id (该特定用户的业务标识id，例如student_no)，
   receiver_role(receiver_id对应的用户所属的角色，例如学生STUDENT)，
   read_status（这里默认设为UNREAD),
   read_time(null),create_time(new Date())。


   其二 如果是发送给特定角色群体的通知，例如发给全体学生的通知，会在notification表插
   入id, notice_id,title, content, type,
   target_type(这里是根据用户的具体角色决定，例如学生就是STUDENT)，
   target_id(这里直接设置为空，和我们是发送给特定角色群体的初衷相呼应）,
   target_scope（某种特定角色设定为SPECIAL_ROLE）,
   create_time(new Date()), update_time(new Date())。

       在notification插入一条数据后，我们还要在notification_receiver插入多条数据，
   id, notice_id(和notification表的notice_id相同，是业务标识id)，
   receiver_id (该特定角色下所有用户的业务标识id，例如学生角色群体的student_no)，
   receiver_role(receiver_id对应的用户所属的角色，例如学生STUDENT)，
   read_status（这里是默认设为UNREAD),
   read_time(null),create_time(new Date())。


   其三 如果是发送全体通知，会在notification表插入id, notice_id,title, content, type,
   target_type(这里设置为空)，target_id(这里设置为空，这和我们是发送给全体通知的设计初衷相吻合）,
   target_scope（群体消息就设为ALL_USERS）,create_time, update_time这些时间字段按当前插入。

        在notification插入一条数据后，我们还要在notification_receiver插入多条数据，
      id, notice_id(和notification表的notice_id相同，是业务标识id)，
      receiver_id (全体所有用户的业务标识id)，
      receiver_role(该receiver_id对应的用户所属的角色，例如学生STUDENT)，
      read_status（这里一般是先设为UNREAD),
      read_time(null),create_time(new Date())。

   然后再来说下读取消息的业务规则：


   ✅ 优化策略一：懒写入（延迟写入 receiver 表）
   当前方式：

   发送时写入所有人的 notification_receiver 数据。

   可选方式：

   notification_receiver 仅在用户第一次读取该通知时再生成该条记录（并设为 UNREAD 或 READ）。

   优点：

   减少 insert 写入压力，尤其是 ALL_USERS 群发场景。

   对消息总量大的系统更友好。

   缺点：

   查询逻辑更复杂，需要判断某用户是否“应该收到”该通知。

   ✅ 优化策略二：通知下发任务异步化
   对于群发和全发场景，发送通知后立即推送异步任务（如 MQ、定时器）来逐批插入 notification_receiver，
   避免长时间阻塞主线程。

【一键报警视频切片业务】
视频切片策略的核心选择问题：是以时间维度划分（如每10秒一个切片），还是以文件大小为维度划分（如每个切片不超过5MB）？这两种方案各有优劣，下面我们通俗对比分析一下：

✅ 一、按“时间”划分（常见，10s一个ts）
原理：
将整个视频按照固定时长（如 10 秒）进行切片，不管文件大小，只关心时间点。

优点：
适用于直播、点播等标准视频播放场景；

兼容性好，配合 .m3u8 可直接用于 HLS 播放；

切片顺序固定、播放顺滑；

支持边下边播（比如播放器只要拿到第一个 ts 文件就可以播放了）；

更易做断点续播、拖动跳转等操作；

很多 FFmpeg 工具链、HLS 服务、CDN 都天然支持这种模式。

缺点：
对于高码率视频，单个切片文件可能会很大；

对于低码率视频，切片文件可能很小，碎片多，影响存储或传输效率；

不易精确控制文件大小（有的需求可能是“每个切片不超过5MB”这种）；

✅ 二、按“文件大小”划分（比如每个切片不超过5MB）
原理：
将视频数据在字节级或帧级进行划分，使每个切片的体积接近某个目标大小（如 5MB）。

优点：
可以精确控制每个切片的体积，便于网络传输（比如用于区块同步、P2P等）；

可避免产生太大或太小的 ts 文件；

适用于离线分发场景，如预加载缓存、磁盘存储优化等；

某些定制系统（如用 chunk 做上传分片存储）会优先考虑这种方式。

缺点：
不适合标准播放器直接播放，因为 .m3u8 和播放器通常以时间为单位；

切片不对齐关键帧可能造成播放卡顿或失败；

编码复杂：需要精准控制码率、GOP 大小，确保切片在关键帧位置结束；

FFmpeg 等工具原生不直接支持这种划分方式，要手动计算大小、拆解二进制内容。

🆚 何时选择哪种方案？
需求场景	推荐方案
视频点播、直播、网页播放	✅按时间划分
希望快速预览视频某段	✅按时间划分
对视频切片体积有严格控制（如10MB以下）	✅按大小划分
文件分发、断点续传、P2P下载	✅按大小划分
与 HLS/CDN 配合播放	✅按时间划分
自定义存储或上传逻辑（如分块上传）	✅按大小划分

如果你的业务是“播放优先”，就用时间切片；如果是“传输或存储优先”，就用大小切片。



【位置消息 视频消息消费业务】
问题：前端发送一条主题为location-topic的消息，现在我需要消费者在消费时调用和位置相关的
业务接口处理location信息。另外 还有一条主题为video-topic的消息，需要消费者在消费时调用
和视频相关的业务接口处理video信息，理论上只需要使用不同topic区分即可，没必要专门新建一个
视频消费者群组和一个位置消费者群组消费，你觉得呢？

-方案解析（同一个 Consumer Group，消费多个 Topic）
定义一个消费者 Group，比如 mainGroup；
这个 Group 订阅多个 Topic，比如 location-topic 和 video-topic；
在消费逻辑中根据 Topic 做分发处理：

consumer.subscribe("location-topic", "*");
consumer.subscribe("video-topic", "*");

consumer.registerMessageListener((msgs, context) -> {
    for (MessageExt msg : msgs) {
        String topic = msg.getTopic();
        switch (topic) {
            case "location-topic":
                locationService.handle(msg);
                break;
            case "video-topic":
                videoService.handle(msg);
                break;
        }
    }
});

✅ 优点：

简洁统一；
初期部署/维护简单；
一个消费者处理多个主题内容。

❌ 潜在问题（建议你特别留意）：

问题类型	描述
职责不清	一个消费者处理多个业务，难以界定职责边界；
代码耦合	消息分发逻辑聚合在一个消费者类里，不利于模块独立维护；
性能问题	一个 Group 的消费压力被两个 Topic 的消息抢占，不易伸缩；
难以扩展	如果后续 video 业务要加重试、监控、限流、消费进度隔离……很难分离；
调试困难	日志混乱，问题排查成本高；
顺序消费风险	多个 Topic 混用同一个 Group 时可能引起消费顺序混乱。

-方案解析（不同 Group，按 Topic 分工）
为每个 Topic 创建一个专属的 Consumer Group：
locationGroup → 只订阅 location-topic
videoGroup → 只订阅 video-topic

java
复制
编辑
// 位置消费者
locationConsumer.setConsumerGroup("locationGroup");
locationConsumer.subscribe("location-topic", "*");

// 视频消费者
videoConsumer.setConsumerGroup("videoGroup");
videoConsumer.subscribe("video-topic", "*");
✅ 优势：

优点	说明
模块清晰	每个 Group 专注一个业务方向，职责清晰；
易于维护	一组消费者只关心一个业务，逻辑集中；
可独立部署	每个 Group 可按需扩容缩容；
消费进度独立	不同 Group 不干扰，进度清晰可控；
后期可加不同策略	比如 video 加限速、location 加实时性保障等；
日志干净	每个 Group 的日志只关注自己业务；


在同一个消费者群组这种方案中，
后续 video 业务要加重试、监控、限流、消费进度隔离时，会很难分离，原因分析如下：
🎯 首先，我们看看你使用“同一个 Consumer Group 消费多个 Topic”时会遇到的问题根源：
consumer.setConsumerGroup("mainGroup");
consumer.subscribe("location-topic", "*");
consumer.subscribe("video-topic", "*");

consumer.registerMessageListener((msgs, context) -> {
    for (MessageExt msg : msgs) {
        if (msg.getTopic().equals("video-topic")) {
            videoService.handle(msg);
        } else if (msg.getTopic().equals("location-topic")) {
            locationService.handle(msg);
        }
    }
});
乍一看很清晰，但未来只要 video 业务变化了，就会变得很难维护和拆分，下面一条条说明为什么：

一、重试机制耦合，无法独立配置
❌ 问题
RocketMQ 的消费重试次数和重试队列是以 Group 为单位管理的。
如果 video 消息消费失败，你想设置：
video-topic 允许重试 10 次；
location-topic 最多重试 2 次；
👉 但这是做不到的，因为整个 Group 的重试策略只能配置一套。

✅ 如果拆成两个 Group
videoGroup 可配 maxReconsumeTimes=10
locationGroup 可配 maxReconsumeTimes=2
你就可以根据业务风险分别处理。

二、消费进度隔离失败
❌ 问题
RocketMQ 的消费进度（offset）是以 Group + Topic + Queue 为维度管理的。
如果你用一个 Group，同时订阅了两个 Topic，它们的 offset 保存在同一个目录下，无法做到
 video-topic 出问题时只暂停 video 消费而不影响 location。

如果你想单独暂停 video-topic 的消费（比如正在升级），那会很麻烦。

三、监控和报警不可控
❌ 问题
如果你用同一个 Group 来消费多个 Topic，RocketMQ 控制台上看到的监控指标会混在一起，比如：
TPS（消费速率）
消费堆积（Lag）
消费失败次数
你会发现这些监控数据无法细分到底是 video-topic 的问题还是 location-topic 的问题，可观测性变差。

四、限流和性能调优困难
❌ 问题
如果未来 video-topic 的消息量暴增，而 location-topic 的量很小：
你想给 video 限流、隔离线程池、隔离消费线程；
但由于使用的是同一个 Group，同一个线程池处理两个 Topic 的消息，
你没法对 video 单独做消费调度控制。
这可能导致 video 抢占资源，location 延迟增加。

五、代码维护与团队协作问题
❌ 问题
随着逻辑增加，你的消费代码变成：

java
复制
编辑
if (topic.equals("video-topic")) {
    if (msg.property("type").equals("streaming")) {
        // ...
    } else if (...) {
        // ...
    }
} else if (topic.equals("location-topic")) {
    // ...
}
维护成本上升、业务边界模糊、容易引发协同混乱

✅ 总结：拆分 Consumer Group 是业务解耦的基础设施能力
问题维度	同一个 Group 的劣势	拆分 Group 的优势
重试策略	一刀切	可定制
消费进度	难以隔离	互不干扰
异常监控	无法精细定位	指标清晰可控
限流控制	不能单独限流	可独立调整
代码维护	if-else 越来越多	模块职责清晰
